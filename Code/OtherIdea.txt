-------------------------------------------------------------------------------
про редактор и конвеер разработки
-------------------------------------------------------------------------------
не нужно пытаться сделать редактор по типу юнити/уе - это неверный подход для моих целей, да и многие известные мне люди не используют редактор юнити, а пишут на скриптовом языке. некоторые создают кастомные редакторы поверх движка, а кто-то модифицирует и движок юнити.
И причина проста - мышкой в редакторе не сделать игровую логику. лишняя работа.

вместо этого нужен другой подход - нужно разделить шаги разработки, на разработку ресурсов, на объединение ресурсов в цельное, и на управление игрой.

разработка ресурсов - тут все просто, меши в 3д пакетах, текстуры в фотошопе...но также нужны редакторы для материала, шейдеров, поведения объектов (упрощенная модель скриптов), анимаций.

Дальше нужен редактор сцен в котором ресурсы собираются в сцену. при этом эта сцена - обычный json файл. редактор сцен позволяет накидать объекты, настроить связи между ними. Кроме того редактор помогает идентифицировать объекты для кода

И собственно управление игрой - это как раз чистый код (возможно некая сокращенная длл для быстрой компиляции в режиме разработки), с которым нужно работать в IDE, при этом этот код сильно связан с текстовым описанием сцены, при запуске из IDE он должен моментально грузить сцену из редактора. Кроме того он должен создавать связь с этим текстовым файлом - например если в редакторе поставили новый домик, код должен сразу получить доступ к этому домику.
